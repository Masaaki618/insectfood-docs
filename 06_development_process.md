# 開発進行方針 | 昆虫食初心者ガイド

このドキュメントはIssueの切り出し方・作業の進め方の方針をまとめたものです。

---

## 1. Issueの切り出し方針

IssueはUI画面・APIエンドポイント・インフラ構築の単位で切り出しました。
1つのIssueが「1つの動く機能」になるよう粒度を揃えています。

1Issueあたりの作業量は**半日〜1日程度**を目安にしており、1つのIssueで実装負荷が高くなりすぎないよう意識して切り出しています。

### フロントエンドの切り出し単位

| 切り出し単位 | 理由 |
|------------|------|
| 基盤系（#1〜#2） | 後続のすべての画面が依存するため先行して実施 |
| 画面単位（#3〜#7） | 1画面 = 1つの機能として独立しているため |
| 品質系（#8〜#9） | 全画面が揃ってから横断的に整備する方が効率的なため |

### バックエンドの切り出し単位

| 切り出し単位 | 理由 |
|------------|------|
| 基盤系（#1〜#4） | 後続のAPI実装が依存するため先行して実施 |
| エンドポイント単位（#5〜#8） | 1エンドポイント = 1つのAPI機能として独立しているため |
| インフラ系（#13〜#14） | 実装完了後にまとめて対応する性質のため |

## 2. 依存マップの活用

各Issueの依存関係を依存マップとして明示的に定義しています。「どのIssueが終わっていないと次に進めないか」が一目でわかるため、作業の段取りミスなく進められる状態を保っています。

```
#1（基盤構築）
  └─→ #2（テーマ・共通スタイル）
        ├─→ #3（昆虫一覧画面）
        │     └─→ #4（昆虫詳細画面）
        └─→ #5（診断フロー画面）
              └─→ #6（診断結果画面）
                    └─→ #7（トップ画面）
                          └─→ #8〜#10
```

## 3. フェーズ構成の方針

まず「動くもの」を最速で作り上げ、アプリとして成り立っていることを早い段階で確認したかったため、Walking Skeleton方式を採用しました。

```
Phase 1（#1〜#2）: 骨格構築
→ とりあえず起動して画面遷移が通る状態を確立する

Phase 2（#3〜#7）: コア機能実装
→ 4画面 + API連携を実装してMVPを動かせる状態にする

Phase 3（#8〜#9）: 品質向上
→ エラー処理・UIの整備をして届けられる品質にする

Phase 4（#10）: リリース準備
→ 本番環境との疎通確認・ビルド検証
```

機能を積み上げながら「アプリとして動いている」という状態を早めに確認できるため、手戻りが少なくなるメリットがあります。

---

## 4. リポジトリ構成の方針

フロントエンド・バックエンドそれぞれのコードを見やすく管理するため、別リポジトリに分けました。設計ドキュメントも独立したリポジトリにすることで、コードと設計書を切り離して管理しています。

```
insectfood-frontend  （Flutter）
insectfood-backend   （Go / Gin）
insectfood-docs      （設計ドキュメント）
```

## 5. 受け入れ基準（AC：Acceptance Criteria）の設計意図

今回はレビュワーを設定していない（Claude APIのコスト面の都合）ため、「何ができたら完了か」の基準を自分で明確にする必要がありました。ACを事前に定義することで実装の迷いをなくし、完了の判断基準を明確にしています。

**AC（Acceptance Criteria）とは**：
- Issueがどのような条件を満たしたら「完了」と見なすか定義したもの
- 各Issueごとに「達成すべき条件」をチェックリスト形式で記載
- これを満たさないと、いくら実装しても「完了」ではない
